# CSRF Protection Documentation

## Overview

Cross-Site Request Forgery (CSRF) protection is implemented automatically by NextAuth.js for the SwiftTriage authentication system. This document explains how CSRF protection works and how it's integrated into the application.

## How CSRF Protection Works

NextAuth.js implements CSRF protection using the **double-submit cookie pattern**:

1. **Token Generation**: When a user visits the application, NextAuth generates a unique CSRF token
2. **Token Storage**: The token is stored in two places:
   - As an HttpOnly cookie (`__Host-next-auth.csrf-token` in production)
   - In the session state (accessible to the application)
3. **Token Validation**: On state-changing requests (POST, PUT, DELETE), NextAuth validates that:
   - The CSRF token in the cookie matches the token in the request
   - The token has not expired
   - The token is cryptographically valid

## Implementation Details

### Automatic Token Generation

CSRF tokens are automatically generated by NextAuth.js when:
- A user first visits the application
- A user signs in
- A token expires and needs renewal

**Configuration**: No additional configuration is required. CSRF protection is enabled by default in NextAuth.js.

### Token Inclusion in Requests

#### Form Submissions

When using NextAuth's `signIn()` method (as in our login form), CSRF tokens are automatically included:

```typescript
// app/login/page.tsx
const result = await signIn('credentials', {
  email: sanitizeInput(email),
  password,
  redirect: false,
});
```

NextAuth's client-side `signIn()` function automatically:
1. Reads the CSRF token from the cookie
2. Includes it in the authentication request
3. Validates the response

#### API Route Protection

For custom API routes that need CSRF protection, NextAuth provides the `getCsrfToken()` helper:

```typescript
import { getCsrfToken } from 'next-auth/react';

// Client-side: Get CSRF token
const csrfToken = await getCsrfToken();

// Include in fetch request
const response = await fetch('/api/custom-endpoint', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
  },
  body: JSON.stringify({
    csrfToken,
    // ... other data
  }),
});
```

### Token Validation

NextAuth automatically validates CSRF tokens on:
- `/api/auth/signin` (login endpoint)
- `/api/auth/signout` (logout endpoint)
- `/api/auth/callback/*` (OAuth callbacks - not used in our credentials-only setup)

**Validation Process**:
1. Extract CSRF token from request body or headers
2. Compare with token stored in HttpOnly cookie
3. Verify token signature and expiration
4. Reject request if validation fails (HTTP 403 Forbidden)

## Security Properties

### Requirements Coverage

This implementation satisfies the following requirements:

- **Requirement 12.5**: CSRF token generation
  - ✅ Tokens automatically generated by NextAuth.js
  - ✅ Cryptographically secure random tokens
  - ✅ Tokens stored in HttpOnly cookies

- **Requirement 12.6**: CSRF tokens included in form submissions
  - ✅ `signIn()` method automatically includes tokens
  - ✅ Tokens available via `getCsrfToken()` for custom forms

- **Requirement 12.7**: CSRF token validation on state-changing requests
  - ✅ Automatic validation on authentication endpoints
  - ✅ Double-submit cookie pattern prevents CSRF attacks
  - ✅ Failed validation returns HTTP 403 Forbidden

### Attack Prevention

CSRF protection prevents the following attack scenarios:

1. **Malicious Form Submission**: An attacker cannot create a form on their site that submits to our authentication endpoints, because they cannot access the CSRF token stored in our HttpOnly cookie.

2. **Cross-Origin Requests**: Even if an attacker tricks a user into making a request from another site, the browser's same-origin policy prevents reading the CSRF token cookie.

3. **Token Replay**: CSRF tokens are time-limited and tied to the user's session, preventing replay attacks.

## Testing

CSRF protection is tested indirectly through our authentication integration tests:

- **Login Tests** (`app/api/auth/[...nextauth]/route.test.ts`): Verify that login requests succeed with valid CSRF tokens
- **Registration Tests** (`app/api/register/route.test.ts`): Verify that registration requests are protected

### Manual Testing

To manually verify CSRF protection:

1. **Valid Request**: Use the login form normally - should succeed
2. **Missing Token**: Remove CSRF token from request - should fail with 403
3. **Invalid Token**: Modify CSRF token value - should fail with 403
4. **Expired Token**: Wait for token expiration - should fail with 403

## Configuration

### Environment Variables

No additional environment variables are required for CSRF protection. It uses the same `NEXTAUTH_SECRET` as the rest of the authentication system:

```env
NEXTAUTH_SECRET=your-secret-key-here
```

### NextAuth Configuration

CSRF protection is enabled by default in our NextAuth configuration (`lib/auth.ts`):

```typescript
export const authOptions: NextAuthOptions = {
  // ... other options
  secret: config.nextAuth.secret, // Used for CSRF token signing
};
```

## Best Practices

### Do's

✅ **Use NextAuth's built-in methods**: Always use `signIn()`, `signOut()`, etc. for authentication operations
✅ **Include CSRF tokens in custom forms**: Use `getCsrfToken()` for any custom authentication forms
✅ **Validate on state-changing operations**: Ensure all POST, PUT, DELETE requests are protected
✅ **Use HTTPS in production**: CSRF protection is most effective over HTTPS

### Don'ts

❌ **Don't disable CSRF protection**: It's enabled by default - keep it that way
❌ **Don't expose CSRF tokens in URLs**: Tokens should only be in cookies and request bodies
❌ **Don't implement custom CSRF logic**: NextAuth's implementation is battle-tested
❌ **Don't store CSRF tokens in localStorage**: Use HttpOnly cookies only

## Troubleshooting

### Common Issues

**Issue**: Login fails with "CSRF token mismatch" error

**Solutions**:
1. Clear browser cookies and try again
2. Verify `NEXTAUTH_SECRET` is set correctly
3. Check that cookies are enabled in the browser
4. Ensure the application is served over HTTPS in production

**Issue**: CSRF token not included in requests

**Solutions**:
1. Verify using NextAuth's `signIn()` method, not custom fetch
2. Check that `getCsrfToken()` is called before making custom requests
3. Ensure cookies are not blocked by browser settings
